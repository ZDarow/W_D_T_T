## 1. Управление сервисом

### Статус

```bash
systemctl status wdtt
```

Успешный запуск выглядит так:
```
● wdtt.service - WDTT — WireGuard over TURN Tunnel Server
     Loaded: loaded (/etc/systemd/system/wdtt.service; enabled; vendor preset: enabled)
     Active: active (running) since ...
   Main PID: 12345 (wdtt-server)
      Tasks: 10 (limit: 2048)
     Memory: 25.2M
        CPU: 142ms
     CGroup: /system.slice/wdtt.service
             └─12345 /usr/local/bin/wdtt-server -listen 0.0.0.0:56000 ...
```

### Запуск/Остановка/Перезапуск

```bash
systemctl stop wdtt      # Остановить
systemctl start wdtt     # Запустить
systemctl restart wdtt   # Перезапустить
systemctl disable wdtt   # Убрать из автозагрузки
systemctl enable wdtt    # Добавить в автозагрузку
```

### Логи в реальном времени

```bash
journalctl -u wdtt -f
```

## 2. Telegram-бот

### Настройка бота

1. Напишите [@BotFather](https://t.me/BotFather) в Telegram
2. Создайте нового бота: `/newbot`
3. Получите токен: `123456:ABC-DEF1234ghIkl-zyx57W2v1u123ew11`
4. Узнайте свой Admin ID: напишите [@userinfobot](https://t.me/userinfobot) — получите число

### Команды бота

| Команда | Описание |
|---------|----------|
| `/start` | Главное меню |
| `/new` | Создать временный пароль |
| `/list` | Список всех паролей |

### Создание пароля

```
1. Отправьте /new
2. Введите количество дней (1-365)
3. Выберите порты (стандартные или свои)
4. Введите VK-хеш
5. Получите готовую ссылку для подключения
```

Формат ссылки:
```
wdtt://<IP>:<DTLS_PORT>:<WG_PORT>:<TUN_PORT>:<PASSWORD>:<VK_HASH>
```

### Управление паролями

Из списка `/list` можно:
- 🔍 **Просмотреть детали** — трафик, устройство, срок действия
- 🗑 **Отвязать устройство** — освободить пароль для другого клиента
- ⏸ **Деактивировать/Активировать** — временно заблокировать пароль
- ❌ **Удалить пароль** — полностью удалить

## 3. Мониторинг и логи

### Статистика сервера

Каждые 10 секунд сервер пишет статистику:
```
[СТАТ] Активных: 1 | Всего: 5 | NAT: MASQUERADE iptables ✅ | ↑0.15 МБ | ↓0.42 МБ
```

Также в `/etc/wdtt/server.log` сохраняется JSON со статистикой:

```json
{
  "active": 1,
  "total": 5,
  "nat": "MASQUERADE iptables ✅",
  "uptime": "1д 2ч 30м",
  "down_gb": "0.42",
  "up_gb": "0.15",
  "passwords": 3,
  "devices": 2,
  "timestamp": 1719560000
}
```

### Ключевые логи

| Сообщение | Значение |
|-----------|----------|
| `[WRAP] OK: ключ выбран для ...` | Клиент успешно аутентифицирован |
| `[WRAP] Отказ: RTP AEAD auth failed` | Неверный пароль или повреждённый пакет |
| `[WG] Новое устройство ... (IP: ...)` | Подключилось новое устройство |
| `[WG] Отказ: пароль ... истёк` | Попытка подключения по истёкшему паролю |
| `[WG] Пароль ... привязан к устройству ...` | Первое подключение устройства |
| `[DB] Удалено истёкших паролей: N` | Автоматическая очистка |
| `[SALAMANDER] Включён` | Salamander-обфускация активна |
| `[JITTER] Включён` | Jitter активен |

## 4. Управление пользователями

### Типы паролей

- **Мастер-пароль** (`-password`) — бессрочный, полный доступ
- **Сгенерированные пароли** (через бота) — временные, до 365 дней

### Ограничения

- Максимум 10 активных сгенерированных паролей
- Каждый пароль может быть привязан только к одному устройству
- Истёкшие пароли удаляются автоматически (проверка раз в час)

### Удаление устройства вручную

```bash
# Показать всех пиров WireGuard
wg show

# Удалить пир
wg set wdtt0 peer <PUBLIC_KEY> remove
```

## 5. Обфускация трафика

### Salamander-обфускация

Включение (в конфиге):
```json
{
  "salamander_key": "aabbccdd11223344aabbccdd11223344"
}
```

Или через флаг:
```bash
wdtt-server -salamander-key aabbccdd11223344aabbccdd11223344
```

**Требования к ключу:**
- Hex-строка, минимум 32 символа (16 байт)
- Максимум — без ограничений (больше = безопаснее, но медленнее)
- Должен совпадать на сервере и клиенте (для будущих версий клиента)

**Как это работает:**
1. Каждый RTP-пакет XOR-ится с криптостойким потоком ключей
2. Ключ генерируется через HKDF-SHA256(мастер-ключ, nonce пакета)
3. nonce — это первые 12 байт RTP-заголовка (SSRC + seq + ts)
4. На принимающей стороне XOR применяется обратно перед AEAD-расшифровкой

### Jitter (рандомизация таймингов)

```json
{
  "jitter": true
}
```

Добавляет случайную задержку 1-10 мс перед отправкой каждого UDP-пакета, нарушая:

- **Timing analysis** — DPI не может определить периодичность отправки пакетов
- **Packet inter-arrival time** — интервалы между пакетами выглядят как реальный VoIP/WebRTC
- **Burst detection** — пакеты не отправляются пачками

### Padding (маскировка размера)

Каждый RTP-пакет дополняется случайным числом байт (1-255) в конце. Размер пакета варьируется, имитируя:

- Opus-аудио: 100-500 байт
- H264-видео: 600-1400 байт  
- SRTP-зашифрованные потоки: 1200-1400 байт

## 6. Советы по производительности

### Оптимальные настройки сервера

```bash
# BBR (уже включено сервером)
sysctl net.ipv4.tcp_congestion_control  # должно быть bbr

# Буферы сокетов
sysctl -w net.core.rmem_max=67108864
sysctl -w net.core.wmem_max=67108864

# Выделенный CPU для обработки пакетов
# (для серверов с несколькими ядрами)
```

### Когда использовать Salamander

- **Включайте, если** трафик явно блокируется DPI
- **Отключайте, если** скорость критична и блокировок нет
- Salamander добавляет ~2-5% накладных расходов на CPU

### Когда использовать Jitter

- **Включайте** при подозрении на тайминговый анализ трафика
- **Отключайте, если** нужна минимальная задержка (онлайн-игры, VoIP)

## 7. Решение проблем

### Сервер не запускается

```bash
# Проверить логи
journalctl -u wdtt --no-pager -n 50

# Типичные ошибки:
# [WG] CreateTUN: ... — нет TUN-устройства
# [WRAP] нет активных паролей для WRAP — не задан -password
# panic: ... — ошибка в конфиге

# Включить TUN:
modprobe tun
mkdir -p /dev/net && mknod /dev/net/tun c 10 200 && chmod 666 /dev/net/tun
```

### Клиент не подключается

1. **Проверьте, что сервер запущен:**
   ```bash
   systemctl status wdtt
   ```

2. **Проверьте порты:**
   ```bash
   ss -ulpn | grep -E '56000|56001'
   ```

3. **Проверьте логи на момент подключения:**
   ```bash
   journalctl -u wdtt --since "30 seconds ago"
   ```
   
   Ищите:
   - `[WRAP] OK: ключ выбран для ...` — аутентификация прошла
   - `[WRAP] Отказ: RTP AEAD auth failed` — неверный пароль
   - `[WG] Отказ: пароль ... привязан к другому устройству` — пароль уже используется

4. **Проверьте firewall:**
   ```bash
   ufw status | grep 56000
   ```

5. **Проверьте IP forwarding:**
   ```bash
   cat /proc/sys/net/ipv4/ip_forward  # должно быть 1
   ```

### Низкая скорость

1. Проверьте, не включён ли Jitter без необходимости
2. Проверьте использование CPU:
   ```bash
   top -p $(pgrep wdtt-server)
   ```
3. Проверьте логи на наличие ошибок NAT
4. Убедитесь, что BBR включён

### Как полностью сбросить сервер

```bash
# Остановить сервис
systemctl stop wdtt

# Удалить TUN-интерфейс
ip link del wdtt0

# Очистить iptables
iptables -t nat -D POSTROUTING -s 10.66.66.0/24 -o $(ip route show default | awk '{print $5}') -j MASQUERADE 2>/dev/null || true

# Удалить конфигурацию
rm -rf /etc/wdtt

# Удалить systemd-юнит
systemctl disable wdtt
rm /etc/systemd/system/wdtt.service
systemctl daemon-reload

# Удалить бинарник
rm /usr/local/bin/wdtt-server
rm /usr/local/bin/wdtt-start.sh
```
