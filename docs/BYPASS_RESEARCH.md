# Исследование методов обхода DPI/блокировок мобильного интернета

> Дата: 28.06.2026
> Контекст: WDTT — WireGuard over TURN Tunnel
> Текущий транспорт: DTLS поверх TURN (UDP)

---

## Сводка найденных техник

### 1. REALITY (XTLS/Xray-core)

**Как работает:** Сервер не имеет своего TLS-сертификата — «отражает» TLS-рукопожатие реального сайта. Клиент использует uTLS-имитацию браузера. При активном зондировании трафик уходит на реальный сайт (fallback).

**Применимость к WDTT:** Средняя. Только TCP. Но техника «отражения» может быть адаптирована для DTLS.

**Референс:** `github.com/XTLS/REALITY`

---

### 2. XHTTP (XTLS) — HTTP/1.1, H2, H3 транспорты

**Как работает:** Универсальный транспорт. `packet-up` (множественные POST на upstream, длинный GET на downstream), `stream-up` (gRPC-маскировка). Header padding (случайные данные в Referer/X-Padding). XMUX — рандомизация мультиплексирования. Разделение up/down через разные протоколы/IP.

**Применимость к WDTT:** **Высокая.** H3 (QUIC) работает поверх UDP. Header padding, рандомизация размеров, разделение up/down — всё применимо к TURN.

**Референс:** `github.com/XTLS/Xray-core`

---

### 3. uTLS (ClientHello Fingerprint Randomization)

**Как работает:** Форк crypto/tls, контролирующий каждый байт ClientHello. Режимы: HelloRandomized, HelloChrome_Auto, HelloCustom. Roller — автоматический перебор отпечатков.

**Применимость к WDTT:** Косвенная. Для DTLS нет аналога uTLS, но принцип рандомизации отпечатков можно адаптировать для QUIC/DTLS.

**Референс:** `github.com/refraction-networking/utls`

---

### 4. Hysteria 2 — QUIC/HTTP3-туннель

**Как работает:** Кастомный QUIC с маскировкой под HTTP/3. Аутентификация через `/auth` с проприетарным кодом 233. **Salamander Obfuscation** — XOR-обфускация QUIC-пакетов с BLAKE2b + pre-shared key.

**Применимость к WDTT:** **Очень высокая.** QUIC + UDP + обфускация — идеальная модель для WDTT. Salamander может быть напрямую заимствован.

**Референс:** `github.com/apernet/hysteria` (22k ⭐)

---

### 5. Salamander Obfuscation

**Формат пакета:**
```
[8 bytes Salt] [Payload XOR-ed с BLAKE2b(key + salt)]
```

**Pros:** Минимальный оверхед (8 байт), статистически неотличим от случайных данных, простая реализация.

**Применимость к WDTT:** **Прямая.** Может быть применён к TURN/DTLS-пакетам.

---

### 6. Фрагментация TCP-пакетов

**Статус на 2026:** Новые DPI (Иран) научились делать полную TCP-реассемблировку — фрагментация больше не работает. Только для TCP.

**Применимость к WDTT:** Низкая. WDTT работает поверх UDP.

---

### 7. WebRTC-туннелирование (VP8-маскировка)

**Как работает:** Упаковка данных в VP8 видео-фреймы — hardcoded VP8-префикс + XChaCha20 nonce + AEAD-шифротекст. Pacing по 10ms, ~6.5 Mbps на один трек.

**Применимость к WDTT:** **Высокая.** WebRTC использует UDP/DTLS — как и WDTT. Можно эмулировать медиа-паттерны (размеры SRTP-пакетов, интервалы).

**Референс:** `github.com/kulikov0/whitelist-bypass`, `github.com/pion/webrtc`

---

### 8. TURN-медиа-маскировка

**Техники:**
- Padding до типовых размеров SRTP (1200-1400 байт)
- Имитация Opus-аудио (20ms интервалы) или H264-видео (33ms, 42ms)
- Подмена DTLS-фингерпринта под браузерный
- Примешивание STUN-сообщений
- TURN Channels с мусорными данными

**Применимость к WDTT:** **Прямая.** WDTT базируется на TURN/DTLS.

## Рекомендации для интеграции в WDTT

### Приоритет 1: Salamander-обфускация для DTLS-пакетов

Наложить XOR-обфускацию BLAKE2b на TURN/DTLS-пакеты:
```go
// Псевдокод — адаптация Salamander для WDTT
func obfuscate(packet []byte, key []byte) []byte {
    salt := make([]byte, 8)
    rand.Read(salt)
    h := blake2b.Sum256(append(key, salt...))
    for i := range packet {
        packet[i] ^= h[i%32]
    }
    return append(salt, packet...)
}
```

**Оверхед:** 8 байт на пакет.

### Приоритет 2: Медиа-маскировка TURN-трафика

- Padding DTLS-записей до 1200-1400 байт (SRTP-размер)
- Фиксированные интервалы отправки (20ms — Opus, 33ms — H264)
- Примешивание STUN-сообщений каждые N пакетов
- Случайный MTU в диапазоне 1280-1472

### Приоритет 3: QUIC как альтернативный транспорт

- Использовать `quic-go` как транспорт вместо/дополнительно к DTLS
- Маскировка под HTTP/3 (Hysteria 2 подход)
- Преимущество: QUIC сложнее заблокировать, чем DTLS

### Приоритет 4: XHTTP-стиль управления соединениями

- Рандомизация параметров мультиплексирования
- Разделение upstream/downstream по разным TURN-серверам
- Header padding в TURN-сообщениях

### Приоритет 5: WebRTC VP8-туннель как fallback

- При блокировке TURN — переход на WebRTC DataChannel
- Упаковка данных в синтетические VP8-фреймы
- Использование whitelist-серверов (VK, Яндекс) как relay

## Ключевой вывод

> Успешный обход DPI — не в одном протоколе, а в **комбинации техник**, которые вместе создают трафик, неотличимый от легитимного на всех уровнях: транспортном (UDP/QUIC), криптографическом (DTLS/обфускация), прикладном (HTTP/3/WebRTC) и статистическом (размеры/тайминги).

### План внедрения

1. **Краткосрочно:** Добавить Salamander-обфускацию и рандомизацию размеров пакетов — минимальные изменения кода, максимальный эффект.
2. **Среднесрочно:** Реализовать медиа-маскировку TURN-пакетов (имитация RTP/SRTP).
3. **Долгосрочно:** QUIC-транспорт (Hysteria 2-подход) или WebRTC VP8-туннель.
